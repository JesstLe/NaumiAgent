# 项目讲解稿

使用方法：先背 30 秒版，再练 3 分钟版。8 分钟版适合项目答辩或深挖面试。

## 30 秒版本

我做的是一个本地 Coding Agent 系统，不是普通聊天机器人。它的核心是一个 Agent Runtime：用户输入任务后，系统会组装上下文，通过模型路由做推理，再通过统一 Tool Registry 调用文件、Shell、浏览器、记忆、任务等工具。工具调用会经过权限和预算控制，结果会回写到上下文和会话存储里。项目还支持 MCP、浏览器自动化、自我审查和目标追踪，重点探索 Agent 怎么从 demo 变成可控、可恢复、可二开的工程系统。

## 3 分钟版本

这个项目叫 NaumiAgent Lite，可以理解成一个本地 Coding Agent 的工程原型。

我当时做这个项目的出发点是：很多 Agent 项目只是在 prompt 里告诉模型「你可以做什么」，但真正落到工程上，会遇到几个问题：模型怎么调用工具、危险操作怎么确认、长任务怎么恢复、历史上下文怎么管理、工具越来越多怎么扩展。

所以我把系统拆成几层。

第一层是入口层。项目支持 CLI、单任务执行和 API 服务。CLI 适合本地开发者使用，API 适合后续接 Web 或其他前端。

第二层是 Agent Runtime。它负责接收用户输入、组装上下文、调用模型、解析工具调用、执行工具、把结果回写给模型。这个过程类似 ReAct，不是一次性生成答案，而是多轮观察和行动。

第三层是工具层。项目把文件、Shell、浏览器、记忆、任务、调度等能力统一成工具，每个工具都有 name、description、parameters schema 和 execute。这样 Runtime 不需要知道每个工具的内部细节，只要通过统一接口调用。

第四层是安全和状态。工具执行前会经过权限和预算控制，危险操作不能让模型绕过。会话历史会存到 SQLite，长期记忆和上下文压缩用于处理长任务。

第五层是扩展能力。项目支持 MCP 接入外部工具，也有 browser daemon、自我审查、自我修改、工具锻造和目标追踪这些能力。这里我会诚实区分：有些是已经可演示的能力，有些是更适合继续二开的方向。

我觉得这个项目最大的收获不是接了某个模型，而是理解了 Agent 应用工程化的关键：模型只是决策器，真正难的是工具协议、安全边界、状态管理和验证闭环。

## 8 分钟版本

### 1. 背景

我做这个项目是因为发现很多大模型应用项目同质化比较严重，尤其是 RAG、客服、问答类项目。它们能展示 API 调用能力，但面试追问工程细节时，很容易停留在 prompt 和向量数据库层面。

所以我选择做一个本地 Coding Agent 系统，把重点放在 Agent Runtime 和工程基础设施上。

### 2. 总体架构

项目可以拆成六层：

- Interface：CLI、TUI、API、WebSocket。
- Runtime：Orchestrator、Planner、上下文组装、模型调用。
- Tool：统一工具接口和工具注册。
- State：SQLite 会话、长期记忆、任务状态。
- Safety：权限、预算、guardrails。
- Extension：MCP、浏览器、subagent、self-evolve、Workbench 方向。

这套分层的核心好处是：模型推理和实际执行解耦。模型不能直接操作文件或系统，只能通过工具协议表达意图。

### 3. Runtime

Runtime 的工作流程是：

```text
用户任务 -> 上下文组装 -> 模型路由 -> 工具调用 -> 权限检查 -> 工具执行 -> 结果回写 -> 下一轮推理
```

这里最重要的是，Agent 不应该只生成一次回答，而应该根据工具执行结果不断修正下一步。这就是 ReAct 思路在工程里的落地。

### 4. Tool Registry

工具层是我认为最关键的部分。每个工具都有统一接口：

- name：工具名。
- description：给模型看的能力描述。
- parameters schema：参数结构。
- execute：实际执行逻辑。

这样新增工具时，不需要改 Runtime 主逻辑。Runtime 只关心工具是否注册、参数是否合法、执行结果如何回写。

### 5. Safety

Agent 系统做文件和 Shell 操作时，安全边界非常关键。项目里有权限、预算和 guardrails 三类控制：

- 权限决定这个工具能不能调用。
- 预算限制 token 和长循环消耗。
- guardrails 用于输入输出风险判断。

如果面试官问为什么要做这些，我会回答：Agent 的风险来自模型输出的不确定性，工程系统要把不确定决策限制在确定性执行框架里。

### 6. Memory

项目里的记忆不是简单把聊天记录拼回 prompt。它分成：

- 会话历史：用于恢复上下文。
- 长期记忆：用于跨会话召回稳定事实。
- 上下文压缩：用于长任务时保留关键信息。

这个设计的价值是让 Agent 更适合长任务，而不是只能做一问一答。

### 7. MCP 和 Browser

MCP 让工具扩展不局限于本地函数。以后接 GitHub、数据库或其他服务，都可以通过协议注册到工具系统里。

Browser 能力让 Agent 可以处理真实网页任务，包括打开页面、观察元素、点击、截图和提取内容。相比纯文本任务，这更接近真实自动化场景。

### 8. Self-Evolve

项目里有 self-review、self-modify、forge、pursuit 这些能力。我会把它们解释成「自我改进雏形」：

- self-review：审查源码和设计问题。
- self-modify：在受限范围内修改代码。
- forge：生成新工具脚手架。
- pursuit：按目标持续推进并验证。

这里不能夸成完全自主进化，正确说法是：项目探索了自我改进闭环，但仍需要人工确认、测试和权限边界。

### 9. 总结

这个项目的重点不是模型效果，而是 Agent 工程化：Runtime、工具协议、权限、记忆、MCP、浏览器、自我审查和验证。它适合展示我对 AI 应用从 demo 到工程系统的理解。

## 被质疑时怎么答

### 质疑 1：这是不是包装过度？

回答：

> 我不会把它说成完全生产级系统。我的定位是本地 Agent 工程原型，重点展示 Runtime、工具、记忆、安全、扩展这些工程模块。哪些能力是可演示的，哪些是二开方向，我会分开说明。

### 质疑 2：是不是用了框架就没技术含量？

回答：

> 项目使用 LiteLLM 这类库做模型统一调用，但 Runtime、工具注册、权限、记忆、CLI/API 和自我修改流程是项目自己组织的。我的重点不是重复造模型 SDK，而是设计 Agent 执行链路。

### 质疑 3：为什么不直接用 LangChain？

回答：

> LangChain 适合快速起步，但我想展示的是 Agent Runtime 的内部设计，比如工具调用生命周期、权限、结果回写、上下文压缩和任务状态。自研一部分核心链路更利于理解工程边界。

### 质疑 4：这个项目怎么证明有效？

回答：

> 对 Agent 项目不能只看模型说完成。我会用命令是否执行成功、测试是否通过、文件是否产生、工具调用日志和运行截图作为证据。后续二开里我也会优先补系统化评测集。

### 质疑 5：自我修改是不是很危险？

回答：

> 是危险的，所以必须限制范围。项目里会保护核心模块，只允许特定目录修改，并要求备份和验证。自我修改不是放任模型改代码，而是在受控边界里探索自动化改进。
