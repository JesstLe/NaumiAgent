# 面试题库

用法：先背思路，再换成自己的项目经历。回答时尽量用「问题 -> 设计 -> 取舍 -> 验证」结构。

## Q01：这个项目一句话是什么？

它是一个本地 Coding Agent 系统，核心不是单次聊天，而是让模型通过受控工具链完成代码、文件、浏览器和任务类操作，并具备会话记忆、权限控制和可扩展能力。

## Q02：它和普通 ChatBot 有什么区别？

ChatBot 主要生成文本；这个项目有 Runtime、Tool Registry、Memory、Safety 和任务状态。模型的输出会变成结构化工具调用，工具结果再回写给模型继续推理。

## Q03：为什么说它不是 LangChain 套壳？

因为项目里有自定义 orchestrator、工具基类、工具注册、权限、记忆、CLI/API、任务和自我修改流程。框架或 LiteLLM 只是辅助，核心执行链路是项目自己组织的。

## Q04：ReAct 在项目里怎么理解？

ReAct 是 Reason 和 Act 的循环：模型根据上下文决定下一步，发起工具调用，系统执行工具并返回观察结果，然后模型继续判断，直到任务完成或需要用户介入。

## Q05：为什么模型不能直接执行 Shell？

直接执行风险太高。需要通过 Tool Registry 和权限层判断命令类型、路径范围、预算和危险操作，必要时让用户确认，降低误删文件或泄露信息的风险。

## Q06：Tool Registry 解决了什么问题？

它把不同工具统一成 `name/description/schema/execute` 接口。这样模型看到的是稳定能力列表，Runtime 可以统一做权限、日志、渲染和错误处理。

## Q07：工具 Schema 有什么价值？

Schema 让模型知道工具需要哪些参数，也让系统能做参数校验。没有 Schema，工具调用容易变成自由文本，难以稳定执行和测试。

## Q08：工具执行结果为什么要结构化？

结构化结果便于 UI 渲染、调试、回放和下一轮推理。比如 diff、文件摘要、错误信息、截图路径都应该有明确字段，而不是全部混成一段文本。

## Q09：项目的模型层怎么设计？

通过 LiteLLM 做统一模型调用，并按 fast、capable、reasoning tier 组织模型。这样可以根据任务复杂度选择不同模型，也方便替换 provider。

## Q10：为什么要做多模型路由？

不是所有任务都需要最强模型。分类、格式化、简单工具选择可以走快模型，复杂代码和规划走强模型，可以降低成本并提升响应速度。

## Q11：Memory 分哪几类？

可以分为会话历史、长期记忆和工作上下文。会话历史保证恢复，长期记忆用于跨会话召回，工作上下文用于当前任务决策。

## Q12：上下文压缩怎么做？

当上下文接近阈值时，保留任务目标、关键决策、工具结果和未完成事项，把冗余对话压缩成摘要，必要时写入长期记忆，避免简单截断导致任务断片。

## Q13：SQLite 在项目里起什么作用？

SQLite 用于本地持久化会话、任务等状态。它适合本地 Agent，因为部署简单、可恢复、便于调试，不需要一开始引入复杂数据库。

## Q14：长期记忆为什么不能只存聊天全文？

聊天全文噪声大、召回成本高。更好的方式是提取稳定偏好、项目事实、决策和经验，以更小的信息单元存储和召回。

## Q15：权限系统有哪些层次？

可以分读操作、写操作、Shell、浏览器、网络、MCP 外部工具等层次。不同层次需要不同确认策略，危险操作应该有更严格的用户确认和日志。

## Q16：预算控制有什么用？

预算控制限制 token、成本和工具执行范围，防止 Agent 在长循环或错误状态下无限消耗资源。

## Q17：guardrails 和权限有什么区别？

权限更偏执行层，决定能不能做某个操作；guardrails 更偏输入输出审查，判断请求和响应是否触碰风险边界。

## Q18：MCP 是什么？

MCP 是 Model Context Protocol，用于让模型通过标准协议发现和调用外部工具。它把工具能力从本地函数扩展到外部服务。

## Q19：为什么接 MCP 比直接写 API 调用更好？

MCP 把工具发现、参数 Schema 和调用方式标准化。以后接 GitHub、数据库、浏览器或内部服务时，可以沿用同一套工具注册和权限流程。

## Q20：浏览器自动化在 Agent 项目里有什么意义？

它让 Agent 能处理真实网页任务，例如打开页面、观察元素、点击、截图和提取信息。相比纯文本问答，更接近真实办公和调试场景。

## Q21：browser daemon 和普通 Playwright 有什么区别？

普通 Playwright 更像一次性脚本；browser daemon 更适合长期会话、任务队列、人工接管、回放和可观测调试。

## Q22：为什么需要运行态面板？

Agent 执行多步任务时，用户需要知道它在做什么、卡在哪里、调用了哪些工具、是否等待权限。运行态面板能降低黑盒感。

## Q23：任务系统解决什么问题？

任务系统把 todo、后台任务、subagent、browser run 等状态集中管理，让长任务可以被查看、回复、中止或恢复。

## Q24：scheduler 在项目里可以怎么用？

可以用于定时提醒、定时检查、周期性任务和后台自动化。对 Agent 来说，调度能力让它从即时问答变成长期助手。

## Q25：subagent 的价值是什么？

subagent 可以把独立任务拆给不同执行单元，主 Agent 负责汇总和协调。但它会增加成本和调试复杂度，所以适合复杂任务，不应过早滥用。

## Q26：为什么说多 Agent 不一定更高级？

多 Agent 会带来通信、状态一致性、成本和错误传播问题。很多任务单 Agent 加工具就够，只有独立性强的任务才适合拆分。

## Q27：self-review 是什么？

self-review 是让 Agent 对源码或设计做结构化审查，找缺陷、风险和改进点。它的价值在于形成反思入口，而不是直接自动改一切。

## Q28：self-modify 为什么危险？

因为 Agent 修改自身代码可能破坏核心运行链路。必须限制可修改目录、保护核心模块、创建备份、运行验证，并让用户确认。

## Q29：forge 工具锻造解决什么问题？

forge 用于根据需求生成新的工具脚手架，并注册到系统里。它体现的是 Agent 扩展能力，但生成结果仍需要验证和人工审查。

## Q30：pursuit 目标追踪是什么？

pursuit 是持续追踪目标直到达成、阻塞或预算耗尽的机制。关键是把目标拆成可验证标准，而不是让模型无限循环。

## Q31：这个项目如何体现工程化？

工程化体现在模块边界、统一接口、权限控制、持久化、测试、调试面板、API 和文档，而不是只靠模型回答效果。

## Q32：项目里最核心的模块是哪几个？

`orchestrator`、`tools`、`memory`、`safety`、`model`、`api` 是核心模块。它们分别对应运行时、能力层、状态层、安全层、模型层和服务层。

## Q33：CLI 和 API 为什么都要有？

CLI 适合本地开发者交互，API 适合集成前端、WebSocket 和外部系统。多入口能体现 Runtime 和 UI 解耦。

## Q34：为什么说 UI 不应该理解太多业务语义？

UI 应该展示状态和收集输入，业务语义应由 Runtime 转成稳定的 UIMessage。否则每新增工具或任务状态都要改多个 UI。

## Q35：如何处理工具执行失败？

需要捕获错误、返回结构化失败结果、记录日志，并把失败信息交给模型决定重试、换方案或请求用户介入。

## Q36：如何防止 Agent 无限循环？

通过最大步数、预算、重复失败检测、成功标准和人工介入机制控制循环。目标追踪类任务还要记录每轮证据。

## Q37：如何评价一个 Agent 任务是否完成？

不能只看模型说完成。要看文件是否存在、测试是否通过、命令退出码、页面是否渲染、用户是否确认等外部证据。

## Q38：为什么需要 debug trace？

debug trace 可以复盘模型输出、工具调用、错误和 UI 事件。Agent 系统很容易黑盒化，trace 是定位问题的重要手段。

## Q39：项目测试应该覆盖哪些地方？

应覆盖工具 Schema、权限判断、模型路由边界、记忆存储、CLI 命令、API 路由、浏览器工具和自我修改保护逻辑。

## Q40：AI 项目怎么做单元测试？

把 LLM 输出隔离掉，测试确定性逻辑，比如参数校验、权限策略、存储读写、工具注册、命令解析和错误处理。

## Q41：如何处理 LLM 不稳定？

把不稳定部分限制在决策层，执行层使用确定性工具；同时增加 Schema、校验、重试、日志和验证命令。

## Q42：这个项目有哪些不足？

完整生产级评测闭环还需要继续加强；部分自我演进能力更适合作为雏形展示；多前端协议和执行层隔离仍可继续重构。

## Q43：如果继续二开，优先做什么？

优先做评测闭环和更清晰的工具协议，因为它们能提升可信度。其次再做 browser daemon 深度集成和 Workbench 治理能力。

## Q44：为什么不一开始做很复杂的多 Agent 市场？

因为基础 Runtime、工具、记忆、权限没稳定时，多 Agent 只会放大复杂度。先把单 Agent 工程链路做扎实更重要。

## Q45：如何把项目讲给后端面试官？

强调配置、API、状态持久化、任务调度、权限控制、日志和测试。把 Agent 讲成一个带模型决策层的后端系统。

## Q46：如何把项目讲给 AI 应用面试官？

强调 ReAct、工具调用、上下文组装、长期记忆、MCP、浏览器自动化和模型路由。重点讲从 LLM 调用到 Agent 工程化。

## Q47：如何把项目讲给平台工程面试官？

强调协议边界、Runtime、Tool Protocol、事件流、运行态、权限、安全、可观测性和扩展性。

## Q48：项目怎么体现可扩展性？

通过 Tool 基类、工具注册、MCP 接入、CLI/API 多入口、scheduler、subagent 和 browser daemon，使新增能力不必重写核心 Runtime。

## Q49：项目怎么体现安全意识？

模型不能直接执行危险操作；工具经过权限检查；预算控制限制资源；自我修改保护核心模块；运行结果可审计。

## Q50：项目怎么体现可观测性？

通过运行态面板、debug trace、任务状态、工具结果、截图路径、日志和测试输出，让用户能看到 Agent 在做什么。

## Q51：为什么选择 Python？

Python 的 LLM、自动化、文件处理和数据生态成熟，适合快速构建 Agent Runtime。未来 UI 或 daemon 可以用 TypeScript、Go 或 Rust 拆分。

## Q52：为什么还需要 Node 终端 UI？

Python 适合 Runtime，但复杂终端 UI 用 TypeScript/React/Ink 生态更自然。多前端也是架构解耦的一部分。

## Q53：Docker 在项目里有什么价值？

Docker 可以降低部署差异，封装运行环境，适合 API 服务和演示。但本地开发仍可直接用 `uv` 启动。

## Q54：项目如何处理配置？

通过 YAML 和环境变量组织模型、工作区、记忆、权限等配置，避免把 API Key 和环境差异写死在代码里。

## Q55：如何说明项目不是堆功能？

按 Runtime、Tool、Memory、Safety、Interface、Extension 六层讲，每个功能都放到对应层里，说明它解决的工程问题。

## Q56：如果面试官质疑项目太大怎么办？

承认项目范围大，然后说明自己重点负责或重点理解的模块，比如 Tool Registry、权限系统、记忆压缩或 browser daemon，并用源码和测试证明。

## Q57：如果面试官问你最难的点是什么？

可以回答工具调用和安全边界。难点不是让模型生成命令，而是让命令在可控、可恢复、可审计的环境里执行。

## Q58：如果面试官问你如何量化效果？

可以说当前重点是工程可用性验证，例如命令能否完成、测试是否通过、工具调用成功率、失败恢复次数。后续会补更系统的评测集和指标。

## Q59：如果面试官问你参考了哪些系统？

可以说参考了 Claude Code、Codex、本地 Agent 工具链和 MCP 思路，但项目重点是自己实现 Runtime、工具协议、记忆、安全和可观测链路。

## Q60：这个项目最大的亮点是什么？

最大亮点是把 Agent 从一次性问答变成一个本地可运行、可扩展、可恢复、带安全边界的工程系统。它能展示 AI 应用开发和后端工程能力的结合。
