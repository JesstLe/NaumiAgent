# 07 CLI 兼容与迁移

## 1. 目标

把新 Terminal UI 提升为唯一默认交互面，同时给已有脚本、Prompt Toolkit 用户和 Textual TUI 留下明确、有限且可测试的兼容路径，避免三个前端长期并行演化。

## 2. 表面分级

| 表面 | 定位 | 新功能策略 |
|---|---|---|
| Node Terminal UI | 默认产品界面 | 全部新交互能力 |
| Prompt Toolkit CLI | 经典兼容模式 | 安全修复和关键协议适配 |
| Textual TUI | 旧版维护模式 | 仅严重缺陷修复 |
| 非交互 CLI | 自动化接口 | 保持稳定并独立测试 |

## 3. 迁移步骤

### 阶段 1：并行可用

切换默认入口，保留显式回退；启动失败页直接提供经典模式。收集无法启动原因和协议错误，不采集对话正文。

### 阶段 2：能力冻结

旧界面显示一次性维护提示；文档、README 和帮助文本统一推荐新 UI。新命令页和完成收据只在新 UI 实现。

### 阶段 3：移除评估

连续两个稳定版本后，根据兼容使用率、阻塞缺陷和自动化依赖决定是否移除 Textual。Prompt Toolkit 可作为无 Node 环境的长期最小回退，但不得复制新页面。

## 4. 兼容约束

- `naumi chat --classic` 的会话格式仍与新 UI 共用 SQLite。
- 旧客户端遇到新消息类型时显示安全文本摘要，不崩溃。
- 新 Bridge 在协议版本范围内兼容旧客户端；超出范围明确拒绝。
- 非交互命令的 stdout 保持机器可读，诊断信息进入 stderr。
- 环境变量和配置参数只保留一个解析实现，由各入口调用。

## 5. 帮助与文案

`naumi --help` 首先说明默认进入交互 UI，并列出经典/旧版回退。旧界面维护提示必须包含替代命令和退出方式，不能每次启动重复打扰已确认用户。

中英文帮助共享命令定义和参数名，只有描述本地化。

## 6. 回滚策略

默认入口切换必须由单一配置开关控制，以便紧急版本回滚；开关不改变数据格式。发生严重启动回归时发布修复版本，将默认临时指向 classic，同时保留新 UI 的显式入口和诊断证据。

## 7. 测试与验收

建立命令矩阵测试：无参数、带配置、classic、legacy、非 TTY、缺 Node、wheel 安装和源码运行。使用同一 SQLite 会话分别从新 UI 与 classic 打开，确认消息可读且不重复执行。

验收要求 README、`--help`、安装包和运行行为一致；用户不需要理解内部前端技术栈就能选择默认或兼容模式。
