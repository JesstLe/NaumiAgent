# Terminal UI 干净启动与本地退出设计

日期：2026-07-14

## 1. 目标

修复新 Terminal UI 的两个同源生命周期问题：

- `/q`、`/quit`、`/exit` 必须立即、安全地退出 UI，不能被当成用户消息进入本地发送队列；
- 直接执行 `naumi` 必须得到全新界面状态，不恢复上次退出时的输入草稿、未确认消息、侧栏、滚动位置或折叠状态；
- 只有用户显式执行 `/resume` 或 `/load <id>`，并收到后端 `session/replayed` 事件后，才恢复该历史会话的 UI 快照。

输入历史仍按工作区持久化，因为它通过 `Ctrl+R` 主动读取，不会改变启动后的可见界面，也不等价于恢复会话。

## 2. 根因

当前 `handleSubmitText()` 不识别退出命令，因此 `/q` 会落入普通消息提交路径，先形成 queued outbox，再被写入 UI 快照。下一次启动可能把它呈现为“发送状态待确认”。

同时 `main()` 在发送 `hello` 之前就读取 `__default__` 快照；后端分配会话 ID 时又按 ID 自动切换快照。该机制无法区分“进程刚启动”和“用户主动恢复”，导致草稿、侧栏和本地视图状态跨启动泄漏。

## 3. 方案比较

### 方案 A：继续自动恢复，但过滤 `/q`

只能修复表面症状；普通草稿、未确认消息和打开的侧栏仍会在新进程出现，不符合用户对全新启动的预期。不采用。

### 方案 B：彻底删除 UI 快照

启动一定干净，但 `/resume` 后无法回到历史会话的阅读位置、折叠状态和侧栏上下文，损失已有能力。不采用。

### 方案 C：显式恢复门控

保留每个会话的 UI 快照，但启动时不应用；只有 `session/replayed` 才应用目标会话快照。正常新会话只保存自己的快照，不读取旧状态。采用此方案。

## 4. 退出命令协议

`handleSubmitText()` 返回结构化本地动作：

```js
{ type: "exit" }
```

`submitComposer()` 在收到该动作后调用现有 `exit()`。退出命令的处理顺序必须早于所有发送逻辑，并满足：

- 大小写不敏感，允许命令两侧空白；
- 只接受完整命令 `/q`、`/quit`、`/exit`，不能误伤 `/query`；
- 不调用 `send("submit")`；
- 不写入输入历史；
- 不创建 outbox 项；
- 仍复用现有 `exit()` 的终端恢复、Bridge shutdown 和进程清理链路。

## 5. 启动与恢复状态机

```text
进程启动
  -> createInitialState()（干净状态）
  -> 仅加载工作区输入历史
  -> hello
  -> 后端分配新 session_id：保存新会话快照，不读取旧快照

用户 /resume 或 /load
  -> 后端 session/replayed
  -> 保存离开会话的快照
  -> 切换 session_id
  -> 读取并应用目标会话快照
  -> 重置历史搜索/补全并跳到最新消息
```

恢复门控以事件类型为准，而不是仅比较 session ID。即使后端在同一个 session ID 上重放，也必须应用该 ID 的快照；普通 `hello`、`session/created` 或其他 ID 变化不得触发恢复。

## 6. 持久化边界

继续保存会话快照，以支持显式恢复，但调整读取时机：

- 新启动：不调用 `applyUiSnapshot()`；
- 新会话 ID：只更新存储目标，不读取；
- `session/replayed`：读取目标快照并应用；
- 退出：保存当前会话快照，但下次普通启动不会自动应用；
- 输入历史：继续独立加载；
- 动画帧、临时权限弹窗等瞬时状态仍不持久化。

未确认 outbox 只在显式恢复时显示为“不确定”，且绝不自动重发。用户可以检查后使用 `/retry`。

## 7. 测试

仅运行 Terminal UI 相关小模块：

- 状态单测：三个退出别名均返回 `exit`，不发送、不入 outbox；近似命令仍正常提交；
- 提交器/进程测试：`/q` 退出码为 0，Bridge 收到 shutdown，未收到 submit；
- 进程测试：第一次退出时留下草稿、侧栏和 queued outbox，第二次普通启动全部不可见；
- 显式恢复测试：`session/replayed` 后恢复目标会话快照，queued outbox 标为不确定但不自动重发；
- 回归：输入历史仍能跨进程通过 `Ctrl+R` 使用；
- `node --check` 与 `git diff --check`。

## 8. 自我审视

- 不能只清空 `/q` 文本，因为其他草稿和侧栏仍会泄漏；恢复门控必须位于进程事件编排层。
- 不能删除 queued outbox 的恢复能力，否则显式恢复时用户会丢失一次可能未被后端确认的操作证据。
- `/q` 是本地 UI 控制命令，不需要后端斜杠路由；把它发送给后端会产生竞态和可见残留。
- 崩溃后的普通重启也按干净启动处理。若用户需要恢复崩溃现场，应显式 `/resume`，这与“不主动 resume 就不恢复”的产品语义一致。
