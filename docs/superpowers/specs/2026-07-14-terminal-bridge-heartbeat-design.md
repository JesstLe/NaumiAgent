# Terminal UI 与 Bridge 心跳设计

## 范围

本功能建立新 Terminal UI 到 Python Bridge 的真实活性检测。它复用现有 JSONL `ping` / `pong` 控制通道，不把普通模型输出当作心跳，也不因模型长时间思考误判后端死亡。

Agent 集群与浏览器工作节点的租约心跳将在对应的高并发执行器中复用相同的健康状态语义；本功能不伪造尚未存在的分布式 worker lease。

## 协议

- UI 在 Bridge `ready` 后启动心跳，默认每 5 秒探测一次。
- 同一时刻只保留一个未决探测，避免后端卡顿时持续堆积写入。
- Bridge 的 `pong` 必须关联原始 request ID，并返回：`ok`、当前是否有活动运行、排队对话数。
- UI 记录往返时间。连续 15 秒未收到关联 `pong` 时进入 `stale`；收到后立即恢复为 `healthy`。
- 计时使用单调时钟；墙上时间只用于展示，不参与超时判定。

## 用户体验

- 正常时不占用底栏空间，避免每 5 秒产生时间线消息或闪烁。
- 无响应时底栏显示 `Bridge: 无响应`，只在健康状态首次转坏时产生一条中文警告。
- 恢复时清除底栏告警，并产生一条简短恢复通知，包含最新 RTT。
- 心跳不会自动杀死或重启 Bridge，因为强制重启可能破坏活动模型轮次、工具调用和未保存状态。
- Bridge 进程真正退出仍走现有的断开处理，立即将本地未确认消息标记为失败并退出 UI。

## 生命周期与清理

- `ready` 重复到达不会创建重复 timer。
- UI 正常退出、fatal error 或 Bridge exit 都停止 interval/timeout，防止进程残留。
- 对迟到、重复、未知 request ID 的 `pong` 做安全忽略并记录调试事件。
- 测试使用注入式单调时钟与 timer，不等待真实 5/15 秒。

## 验证

- 独立 Node 单元测试覆盖启动、单飞、超时、恢复、重复启动和停止。
- reducer/footer 测试覆盖 stale 与恢复展示。
- 进程测试使用会响应心跳与故意丢弃心跳的 Bridge fixture，验证真实 JSONL 往返和无响应提示。
- Python Bridge 测试验证关联字段与活动/队列事实。
- 只运行心跳、协议、底栏相关定向测试。
