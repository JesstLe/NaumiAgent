# Codex-Grade Chat Workflow for NaumiAgent Workbench

## 1. 目标

在保留 Workbench 顶部导航和三栏框架的前提下，把 Chat 页面升级为完整的 Codex 式研发对话工作流。升级不止是视觉换皮：消息、执行轨迹、审批、文件变更、Git、后台进程、来源、Mission 和 Issue 必须通过真实数据贯通。

默认语言为中文，同时完整支持英文。实现继续采用 SwiftUI 原生组件，并复用现有 `DaemonController`、Workbench API、流式事件和治理数据，不建立平行状态系统。

## 2. 产品边界

### 2.1 保留

- Workbench 现有顶部导航、路由和三栏页面骨架。
- 当前会话、Mission、Issue、风险等级、并行模式和审批机制。
- 本地服务、SQLite 持久化、WebSocket/SSE 流式事件和现有安全边界。
- `WorkbenchComponentTheme` 确立的安静、紧凑、原生 macOS 组件语言。

### 2.2 本次新增

- Codex 式无气泡助手正文和轻量用户消息块。
- 可折叠执行轨迹、耗时、工具、子 Agent、审批和失败恢复。
- 文件变更摘要及进入 Review 页的真实联动。
- 真实 Git 状态、后台进程和来源引用检查器。
- 悬浮式组合输入器，承载附件、Mission、Issue、风险、并行模式、停止、重试和发送。
- 对话事件到 Workbench 审计事件的可追溯关系。

### 2.3 不做

- 不展示模型内部思维链或隐藏系统提示。
- 不把完整工具参数、密钥、令牌或敏感环境变量放进 UI。
- 不伪造文件变更、Git 状态、后台进程、来源或测试结果。
- 不复制 Codex 的品牌、专有图标或整体应用布局。
- 不改造其他顶部导航页面，除非 Chat 的真实跳转需要补充入口。

## 3. 页面信息架构

页面继续使用左、中、右三栏。三栏在 Workbench 的统一 1440 pt 设计画布上共同缩放，任何支持窗口尺寸下都保持可见；栏内内容独立滚动。

### 3.1 左栏：会话与治理上下文

左栏使用扁平分组列表，不使用嵌套卡片。

1. 当前会话：标题、连接状态、会话 ID 和最近活动时间。
2. 关联 Mission：当前 Mission 及切换入口。
3. 开放 Issue：按风险排序，显示状态点、标题、风险和负责人。
4. 最近运行：显示最近完成、失败或等待审批的对话运行。

选择 Issue 或运行时，中栏滚动到相关消息或执行轨迹，右栏同步显示对应检查器内容。

### 3.2 中栏：对话与执行主线

中栏是主要工作区，正文最大阅读宽度为 760 pt，周围留出稳定边距。

- 用户消息：靠右的浅灰消息块，最大宽度约正文的 72%，圆角不超过 8 pt。
- 助手消息：无背景正文，支持可选择文本、Markdown、列表、代码块和可点击文件路径。
- 元信息：角色和时间弱化显示，不重复占据视觉层级。
- 执行轨迹：在助手结果前使用细竖线和步骤节点表达准备、分析、工具、审批、子 Agent、整理和完成。
- 结构化结果：文件变更、任务创建、测试、命令和子 Agent 使用专用摘要组件。
- 消息动作：复制、赞同、反对、展开、进入审查仅在悬停或选中时显示。

### 3.3 右栏：环境检查器

右栏采用 Codex 式单一检查器容器，由分隔线划分以下区域：

1. 环境信息：服务连接、工作区路径、当前分支和运行模式。
2. 变更：真实文件数量、增删行和脏工作区状态，点击进入 Review。
3. 本地工作区：当前 Worktree、分支、Ahead/Behind 和保留状态。
4. 后台进程：命令安全摘要、PID、状态、启动时间和停止入口。
5. 关联对象：Mission、Issue、审批和验证记录。
6. 来源：文件、截图、任务、执行记录及外部来源引用。

没有真实数据的区域显示紧凑空状态，不显示占位数字。

## 4. 组合输入器

输入器固定在中栏底部，悬浮于内容上方，保持 8 pt 圆角和轻量边框，不使用大面积阴影。

### 4.1 主体

- 3 至 8 行自适应文本输入。
- `Return` 发送，`Shift+Return` 换行；输入法候选期间不误发送。
- 发送中按钮切换为停止；失败后切换为重试。
- 空输入时发送不可用，并提供可访问性原因。

### 4.2 底部工具条

- 添加来源：选择文件、截图或现有 Workbench 对象。
- 任务联动：关闭、创建新 Issue、关联现有 Issue 三种状态。
- Mission：显示并切换当前 Mission。
- 执行策略：风险等级和并行模式收纳在菜单中。
- 发送/停止：使用熟悉的 SF Symbol 图标和 Help 文案。

任务详细字段只在启用“创建新 Issue”时以内嵌抽屉展开，不再长期占用整个右栏。

## 5. 执行轨迹

### 5.1 状态模型

每个用户回合对应一个 `ChatRun`，至少包含：

- `id`、`session_id`、`user_message_id`、开始/结束时间和总耗时。
- 当前阶段：preparing、analyzing、tool、approval、subagent、composing、completed、failed、cancelled。
- 可公开步骤列表，每个步骤有安全标题、状态、耗时和可选结果摘要。
- 关联的 Issue、Mission、文件变更、审批、验证和审计事件 ID。

运行中轨迹默认展开；完成后默认折叠为“已处理 12m 7s”，用户可展开查看。简单且未调用工具的短回复可以只显示轻量耗时行。

### 5.2 工具与审批

- 工具步骤只显示归一化名称、目的摘要、状态和耗时。
- 原始参数在服务端生成安全摘要后才允许进入事件流。
- 需要确认时，轨迹暂停并显示风险、原因、“允许一次”和“拒绝”。
- 审批完成后原地更新同一步骤，不新建重复卡片。
- 子 Agent 展示角色、任务摘要、状态和最终结果，不展示其内部推理。

### 5.3 失败与恢复

- 失败停留在具体步骤，展示可理解的中文或英文错误。
- 可重试步骤只重试安全的失败边界，不重复已成功副作用。
- 取消运行后保留已有步骤和审计记录。
- 连接中断时显示重连状态；恢复后通过运行 ID 补拉事件，避免重复步骤。

## 6. 结构化结果组件

### 6.1 文件变更摘要

展示文件总数、增加/删除行数和最多三条文件路径；其余文件折叠。点击“审查”切换顶部导航到 Review，并选中对应运行或 Issue。

### 6.2 命令结果

展示命令安全摘要、退出码、耗时和截断后的输出。完整日志保存在本地审计记录中，并通过明确动作打开，不在消息正文无限展开。

### 6.3 任务与 Mission

展示创建/关联结果、标题、风险、状态和 ID。点击后跳转任务市场或 Dashboard 的对应对象。

### 6.4 验证结果

展示运行名称、通过/失败数量、耗时和失败摘要，并能进入 Review 或 Timeline 查看完整记录。

## 7. 数据与 API 设计

现有消息历史继续作为模型上下文和最终文本的来源；新增运行与产物模型，不把结构化状态塞进消息正文。

### 7.1 新增服务模型

- `ChatRunRecord`：回合级生命周期和关联对象。
- `ChatRunStepRecord`：可公开执行步骤。
- `ChatArtifactRecord`：文件变更、命令、验证、任务、来源等结构化结果。
- `BackgroundProcessRecord`：由 Workbench 启动或跟踪的进程。
- `SourceReferenceRecord`：本地文件、截图和治理对象引用。

所有记录按 session 隔离并持久化。服务重启后，历史运行、变更、来源和后台进程状态可以恢复或明确标记为已结束/未知。

### 7.2 接口

- 现有流式对话接口继续发送消息 token 和权限事件。
- 新增统一的 run/step/artifact 事件，携带稳定 ID 和递增序号。
- 新增会话运行列表、运行详情、产物列表、环境摘要和后台进程列表接口。
- 文件变更和 Git 摘要由服务端读取真实仓库生成。
- 客户端首次进入页面获取快照，随后按序号消费增量事件；断线后从最后序号恢复。

### 7.3 安全边界

- 服务端负责参数脱敏、路径归一化和输出截断，客户端不承担秘密识别。
- 只允许展示当前注册工作区内的文件和进程。
- 停止后台进程、丢弃变更和执行高风险命令继续走现有审批机制。
- 来源引用持久化路径和类型，不复制无关文件内容。

## 8. SwiftUI 组件边界

`ChatView` 只负责三栏编排和跨组件选择状态，避免继续扩张为单文件页面。

- `ChatContextRail`：左栏会话与治理列表。
- `ChatConversationView`：消息、轨迹、结果和滚动行为。
- `ChatMessageRow`：用户/助手消息呈现。
- `ChatRunTimeline`：运行折叠、步骤和权限决策。
- `ChatArtifactView`：按产物类型分发摘要组件。
- `ChatComposer`：输入、来源和任务联动。
- `ChatInspector`：右栏环境与关联对象。
- `ChatPresentation`：纯值类型的展示策略、格式化和安全可见性判断。

组件通过 DTO 和命令闭包交互，不直接复制网络请求或持久化逻辑。

## 9. 视觉规范

- 使用系统字体，正文 13–15 pt；紧凑面板标题 12–13 pt。
- 字间距保持 0，不随窗口宽度缩放字体。
- 主画布使用白色，栏位使用系统分组背景；仅选中状态和主动作使用蓝色。
- 卡片圆角不超过 8 pt；只为审批、文件变更、验证和错误等真正结构化对象使用卡片。
- 不使用渐变、装饰光斑、彩色边框阵列或页面级悬浮卡片。
- 所有固定格式控件使用稳定尺寸，动态内容不得推动顶部导航或三栏宽度。

## 10. 本地化与可访问性

- 所有新增文案进入 `AppStrings.Chat`，中文为默认，英文逐项对应。
- 图标按钮提供 Help、accessibility label 和禁用原因。
- 颜色不是唯一状态编码；状态同时有图标和文字。
- VoiceOver 顺序为左栏、消息主线、输入器、右栏；执行轨迹按时间顺序朗读。
- 文本支持选择；文件路径和结构化结果支持键盘聚焦。

## 11. 验证策略

### 11.1 单元测试

- 消息角色、空消息、Markdown 和文件路径呈现策略。
- 运行阶段、耗时、折叠状态和安全工具摘要。
- 输入器发送、换行、任务联动、停止和重试状态。
- 环境、Git、后台进程和来源空状态与真实状态。
- 中英文新增文案覆盖。

### 11.2 API 与持久化测试

- run/step/artifact 创建、排序、会话隔离和重启恢复。
- SSE/WebSocket 重连、序号去重和权限决策续跑。
- Git、进程和来源只返回当前工作区真实数据。
- 敏感参数、越界路径和过长输出不会进入 UI DTO。

### 11.3 真实场景验收

1. 普通问答：显示简洁回复和轻量耗时。
2. 工具调用：显示展开轨迹、审批、运行、结果和总耗时。
3. 子 Agent：显示委派和最终结果。
4. 文件修改：显示真实增删行并进入 Review。
5. 任务联动：同一回合创建 Issue 并可追溯。
6. 服务重启：历史运行和产物恢复，无重复步骤。
7. 中文和英文分别运行。
8. 桌面与最小支持窗口截图中三栏始终可见，无重叠和裁切。

## 12. 实施阶段

1. 拆分 Chat 组件并完成 Codex 式消息主线和输入器。
2. 建立 ChatRun、步骤事件和持久化恢复。
3. 接入审批、工具和子 Agent 轨迹。
4. 接入文件变更、Review 跳转和验证摘要。
5. 接入环境、Git、后台进程和来源检查器。
6. 完成任务/Mission/附件联动、本地化和可访问性。
7. 真实应用逐场景验收、截图对照、打包和签名检查。

每个阶段独立测试、独立提交并及时推送。小阶段只运行定向测试；完成一个大模块后运行完整 Swift 测试和相关后端测试。

## 13. 完成标准

- Chat 页面在保留顶部导航和三栏框架的同时，达到 Codex 式信息层级和交互体验。
- 对话、运行、审批、文件、Git、进程、来源、Mission 和 Issue 均使用真实数据。
- 用户可以理解系统正在做什么、为什么暂停、完成了什么以及下一步可以做什么。
- 不展示内部思维链、秘密或未脱敏工具参数。
- 所有新增行为有中英文文案和自动化测试。
- 真实应用完成普通对话、工具、子 Agent、文件变更和任务联动验收。
- 打包产物签名有效，工作区清洁，变更已提交并推送。
