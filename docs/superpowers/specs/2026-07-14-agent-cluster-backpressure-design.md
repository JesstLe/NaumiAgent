# Agent Cluster Backpressure Design

## 状态

- 日期：2026-07-14
- 状态：已实现
- 范围：只处理子 Agent 集群并发；工具批次与浏览器 run 保持各自调度器

## 问题

工具调用已有 `max_parallel_tools` 和写操作串行屏障，但
`SubAgentManager.execute_parallel()` 会为整个列表一次性创建协程。宽 DAG 或多个同时
提交的并行批次会把模型连接、内存和供应商速率限制瞬间打满，也没有队列背压。

## 设计

`safety.max_parallel_agents` 默认 4，可配置 1 至 32。`SubAgentManager` 使用两层有界调度：

1. 每个批次只创建 `min(limit, task_count)` 个 worker，不为所有排队任务创建协程；
2. 所有同时运行的批次共享一个 Semaphore，保证整个 manager 的活跃 Agent 不超过上限；
3. worker 按输入索引领取任务，形成 FIFO 背压；结果仍按原始顺序返回；
4. 普通子任务异常只转换对应的 error 结果，不取消兄弟任务；
5. 父级取消会取消活跃 worker，尚未领取的任务不会启动。

`/runtime subagent` 显示“活跃/上限”和排队数；排队计数在正常完成、异常和父级取消后
都会归零，避免给用户留下虚假的忙碌状态。

`execute_dag()` 继续按依赖层调用同一个有界 `execute_parallel()`，因此宽 DAG 自动受相同
上限约束。设置为 1 时是可预测的串行诊断模式。

## 已验证

- 六个任务、并发 2：首波只启动 0/1，峰值为 2，最终结果顺序稳定；
- 两个并行批次共享同一个上限，合计峰值仍为 2；
- 单任务异常不影响兄弟任务；
- 父级取消只取消两个活跃 worker，其余十八个任务未启动；
- 配置拒绝 0 和 33，接受 1、4、32。

## 后续边界

- 浏览器 run 的隔离与配置化并发属于下一独立切片；
- 工具批次已有独立 `max_parallel_tools`，本功能不改变其写屏障和权限行为；
- 供应商级动态限流、重试抖动与跨进程配额需要后续独立设计，不能由本地 Semaphore 冒充。
