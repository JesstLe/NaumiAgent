# 排队对话立即发送设计

## 目标

允许用户在模型仍在工作时继续按 Enter 发送普通对话。Bridge 必须立即接收消息、反馈队列位置，并在当前运行结束或被取消后按发送顺序自动执行，用户不再需要等待或使用 `/retry`。

## 行为契约

- 普通 `submit` 使用 Bridge 持有的有界 FIFO 队列，容量为 20 条待执行消息。
- 空闲时提交立即开始；忙碌时提交立即进入队列，并依次发出 `user/message`、`run/queued` 和最新 `runtime/status`。
- `run/queued` 包含原始任务文本、从 1 开始的队列位置以及当前队列长度。前端将消息显示为“已排队 · 第 N 位”，并视为已被 Bridge 接收，不保留在本地重试 outbox。
- 当前运行无论成功、失败或被用户取消，Bridge 都自动启动下一条排队对话。共享的 AgentEngine 同一时刻最多执行一轮，保证会话历史有序。
- 队列满时返回关联到本条消息的 `queue_full` 错误；由于消息未被 Bridge 接受，不发出 `user/message`。
- `/q`、其他斜杠命令和权限/交互回复继续走即时控制通道，不进入对话队列。
- `task_submit` 保持互斥。本功能只改变普通对话，不把 Workbench 任务与聊天轮次混合排队。
- 关闭 Bridge 时，所有尚未启动的条目收到 `run/cancelled`，原因说明为界面关闭；它们不会在下次启动时自动恢复。

## 状态与协议

- 新增服务端事件 `run/queued`。
- `runtime/status.tasks.queued_conversations` 暴露当前等待数量，底栏任务摘要使用现有折行机制呈现。
- 前端用户消息新增 `scheduled` 发送状态；收到 `run/started` 后切换为 `accepted`。
- `scheduled` 是服务端已确认状态，不属于传输失败恢复集合，也不会写入本地 outbox。

## 并发与竞态

- 队列和活动任务只在 Bridge 所属 asyncio 事件循环中变更，不跨线程共享。
- 每轮执行结束时由统一的收尾路径清理活动上下文并调度下一轮，避免两个完成回调同时启动任务。
- 取消操作先等待活动 task 确认取消，再发 `run/cancelled`，随后推进队列。
- shutdown 先标记 Bridge 已关闭并清空队列，再取消活动任务，防止收尾回调重新启动排队项。

## 验证

- Python 单元测试覆盖 FIFO、队列满、失败后推进、取消后推进、关闭清队列和状态计数。
- Node 状态测试覆盖 `run/queued`、`run/started` 状态转换和 outbox 排除。
- 终端进程测试真实输入两条消息，观察第二条排队且自动开始，确认没有 `/retry` 和重复气泡。
- 仅运行上述定向测试与相关 Ruff 检查，不运行全量测试。
