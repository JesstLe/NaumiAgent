# NaumiAgent 文档治理与目录整理设计

日期：2026-07-14

## 1. 目标

在不删除旧 Prompt Toolkit CLI 源码的前提下，让仓库文档准确表达当前产品行为，并让未来的
入口、配置和兼容策略变更能够被机器检查，而不是继续依赖人工全文搜索。

本期只处理文档治理这一项独立能力：

- 建立唯一的当前文档入口；
- 把当前事实、产品规格、历史工程记录、迁移资料和参考资料明确分层；
- 修正仍把 `--classic`、手动 legacy fallback 或旧模块启动命令当成现行用法的文档；
- 为全部 Markdown 文档建立可验证的分类覆盖；
- 检查本地链接和当前文档中的退役命令；
- 保留旧 CLI 实现、测试、必要依赖和相关历史设计记录。

不在本期实现语义着色、加载动画或新的跨平台运行时能力。它们分别独立设计、测试和提交。

## 2. 当前审计证据

仓库当前共有 150 余份 `docs/**/*.md`。现有 `docs/README.md` 的标题仍是 2026-05-12 的
“通用智能 Agent 设计方案”，实际只索引早期架构章节和 Mac Workbench，不能作为当前项目总入口。

全文扫描发现：

- 当前 Terminal UI 产品文档仍公开 `naumi chat --classic`；
- 部分交付材料仍推荐 `python -m naumi_agent.main chat`；
- Textual 仍被描述为需要用户手动选择的“旧版维护模式”，与目前自动 fallback 的运行行为不符；
- `docs/superpowers/specs/` 与 `docs/superpowers/plans/` 保存了实施时的历史命令和阶段判断，
  这些内容是工程证据，不应被改写成今天的事实；
- 精确内容哈希扫描没有发现可直接删除的重复 Markdown 文档。

因此，目录整理的关键不是批量移动或删除，而是给文档增加明确语义和可执行门禁。

## 3. 方案比较

### 方案 A：批量重写全部文档

优点是表面上所有文字都能统一。缺点是会篡改历史计划、审计与迁移证据，也容易把尚未实现的
设计错误写成现状。风险不可接受。

### 方案 B：立即搬迁并删除旧文档

优点是目录看起来简洁。缺点是 150 余份文档存在大量相对链接和实施记录，当前也没有重复文件
可作为安全删除依据。大规模移动会制造无价值 diff 和失效链接。

### 方案 C：状态分层、当前入口、机器门禁

重写 `docs/README.md` 为总入口；使用治理清单覆盖所有文档；只更新被标记为当前事实的文档；
历史计划、设计和迁移记录保持原文，并在目录说明中明确“不可作为当前运行说明”。通过脚本检查
分类覆盖、本地链接和退役命令。

采用方案 C。它既整理信息架构，又保留工程证据，并能持续阻止文档重新漂移。

## 4. 文档状态模型

每份文档必须且只能落入以下一种主要状态：

| 状态 | 含义 | 是否必须匹配当前运行行为 |
|---|---|---|
| `current` | 用户入口、安装、配置、运行和当前架构说明 | 是 |
| `product_spec` | 已批准的产品或界面规格，允许包含未完成验收项 | 命令面必须注明实现状态 |
| `historical` | 实施计划、设计决策、审计、优化日志 | 否，保留当时语境 |
| `migration` | 特定系统或版本迁移记录 | 否，必须有迁移范围 |
| `reference` | 外部参考稿、示例、销售或学习材料 | 否，不得冒充当前手册 |
| `quality` | 验收、测试、发布门禁 | 与其声明的目标版本一致 |

治理清单使用按目录/文件匹配的规则，不在 150 余份文件中逐一复制元数据。检查器必须枚举真实
文件，确保每份 Markdown 至少命中一条规则，并拒绝同优先级冲突。

## 5. 规范文档入口

`docs/README.md` 改为唯一文档地图，首屏说明：

- 当前默认界面是 Node Terminal UI；
- 启动失败自动 fallback 到 Textual，显式入口为 `naumi tui`；
- 旧 Prompt Toolkit CLI 代码保留，但无公共启动入口；
- `.naumi/config.yaml`、`.naumi/providers.json` 和 `.naumi/data/` 是项目状态布局；
- “当前使用”“产品规格”“架构历史”“质量与迁移”“参考资料”分别从哪里进入。

根 `README.md` 继续承担安装与快速开始，不复制整套文档地图。

## 6. 当前文档修订范围

本期直接修订：

- `docs/README.md`：改成当前文档地图；
- `docs/product/terminal-ui/01-default-entry-and-runtime-shell.md`：写明已实现的默认入口和自动 fallback，
  移除 `--classic`；
- `docs/product/terminal-ui/07-cli-compatibility-and-migration.md`：把 Prompt Toolkit 定义为“保留源码、
  无公共入口”，把 Textual 定义为受支持 fallback；
- `docs/product/naumiagent-lite/README.md`、课程和交付检查清单：使用 `naumi`/`naumi tui`，不再推荐
  Python 模块内部入口；
- `docs/superpowers/README.md`：说明 plans/specs 是按时间保存的工程记录，不是当前用户手册。

早期架构章节、历史实施计划、迁移日志和来源审计不做全文改写。它们由治理状态和文档地图提供
语境，避免丢失当时证据。

## 7. 机器化门禁

新增 `scripts/check_docs.py` 和小模块测试。检查器执行真实静态扫描：

1. 读取治理清单并验证 Schema；
2. 枚举 `README.md` 与 `docs/**/*.md`；
3. 确认每份文档有唯一有效分类；
4. 解析 Markdown 中的本地链接，忽略代码围栏、HTTP、邮件和纯锚点，验证目标文件存在；
5. 只对 `current` 和明确标为现行的产品文档检查退役入口；
6. 输出中文、逐文件、可行动的错误；成功时报告扫描数量和分类统计。

退役入口规则至少覆盖：

- 将 `naumi chat --classic` 当作可运行命令；
- 将 `python -m naumi_agent.main chat` 当作用户启动方式；
- 将 `naumi ui --legacy` 当作首选手动 fallback，而非弃用别名说明。

历史目录允许出现这些字符串，因为它们记录的是当时设计。门禁不能通过全仓库粗暴字符串禁用
来伪装治理完成。

## 8. 测试与真实验证

遵循小模块验证，不运行全量测试：

- 单元测试用临时文档树覆盖：未分类、冲突分类、有效/失效链接、代码围栏、外部链接、退役命令、
  历史记录豁免和中文错误信息；
- Ruff 只检查 `scripts/check_docs.py` 与对应测试；
- 对真实仓库运行 `python scripts/check_docs.py`，确认全部文档被分类且本地链接有效；
- 用 `rg` 复核当前入口文档不再推荐退役 CLI；
- 检查 `git diff --check`。

## 9. 边界与自我审视

- 不删除旧 Prompt Toolkit CLI 源码、测试或依赖；文档整理不能扩大为代码清理。
- 不改写历史记录中的旧命令，否则会破坏审计价值。
- 不为“目录整洁”制造大规模文件移动；只有发现真实重复、孤立或错误分类时才单独处理。
- 首期链接检查只验证目标文件，不验证标题锚点；锚点在不同 Markdown 渲染器间规则不同，后续可在
  有统一渲染器时加入。
- 产品规格可以描述目标态，但必须和当前手册分开，避免用户把尚未验收的内容当成现有能力。

验收完成后，这一期独立提交并合入 `main`，再进入语义着色功能。
