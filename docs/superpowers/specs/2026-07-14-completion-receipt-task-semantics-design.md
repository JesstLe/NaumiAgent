# Completion Receipt Task Semantics Design

## Goal

让完成回执成为一次运行的可信结论，而不是 Git 状态转储：任务已经成功时显示“已完成”，验证证据紧邻结果，任务影响与 Naumi 自身运行日志分开，失败和待处理项仍保持醒目。

## Root Cause Evidence

当前 `RunReceiptBuilder` 只要看到任意文件差异且没有 pytest/ruff 等命令，就把运行判成 `partial`。这把“任务是否完成”和“代码交付是否经过测试”混成了一个维度。删除目录这类操作虽然工具返回成功、助手正文也确认成功，仍会被回执改判为“部分完成”。

`diff_run_changes()` 对运行前存在、运行后消失的路径统一标记为 `restored`。对于运行前的未跟踪文件，这个变化实际是本轮删除，却在 UI 中显示成“还原”。

`bash_run` 的命令参数没有参与文件影响归因；`.naumi/terminal-ui-debug.jsonl` 与 `data/debug-runs/` 等 Naumi 运行证据也和用户任务改动混排。因此截图中的删除任务同时出现错误状态、错误动词、后台日志和通用“验证/审查/提交”建议。

## Chosen Semantics

### 1. 任务结果与工作区背景分离

`ReceiptChange` 增加向后兼容的 `scope` 字段：

- `task`：由本轮工具直接产生或属于普通工作区文件的变化。
- `background`：Naumi 在运行期间写入的调试轨迹，不参与任务结果和验证风险判定。

协议仍保持 schema version 1；缺少 `scope` 的旧回执默认按 `task` 处理。新 UI、TUI 和前端协议归一化器同步支持该字段。

仅把已知运行证据路径归入后台：`.naumi/terminal-ui-debug.jsonl` 与 `data/debug-runs/`。若工具明确把这些路径作为目标，则以任务归因为准，避免隐藏用户主动操作。

### 2. 删除操作使用确定性后置条件

对 `file_delete` 及可安全解析的 `bash_run rm ...`：

1. 从结构化工具参数或 shell token 中提取工作区内的字面量目标；忽略通配符、变量、命令替换和工作区外路径。
2. 成功返回后直接检查目标是否不存在。
3. 生成一条文件系统验证证据，内容为“路径已不存在: <path>”。
4. 若退出码非零或目标仍存在，记录失败证据并阻止回执宣称完成。

这不是相信模型文字，也不是 Prompt 套壳；结论来自工具事件、真实退出码和文件系统状态。

### 3. Git 变化使用用户可理解的动词

运行前为 `untracked`、运行后消失的路径标记为 `removed_untracked`，新 UI 与 TUI 显示“删除”。真正把已跟踪脏文件恢复到 HEAD 的场景仍显示“还原”。

`removed_untracked` 和 `restored` 不产生“提交改动”建议，因为运行结束后没有对应 Git 差异可提交。代码修改、跟踪文件删除等可提交变化仍保留验证、审查和提交建议。

### 4. 回执默认只展示结论和可行动证据

新 UI 与 TUI 采用相同层级：

- 第一行：结果与耗时。
- 第二行：助手的单行摘要。
- 验证：最多两条真实验证结果；没有验证且没有待验证的任务改动时不显示“0/0”。
- 影响：任务变化按状态聚合，避免逐文件重复刷屏。
- 后台：只显示“工作区另有 N 项运行时变化”，不展开轨迹文件。
- 审批：成功/bypass 审批默认隐藏；拒绝或错误才显示。
- 风险与下一步：仅在确实存在时显示。

Git 分支/脏状态不再占据每张成功回执的固定行；只有 Git 不可用、同步落后或存在可提交的任务变化时才作为可行动信息出现。

## Outcome Rules

- 取消、引擎失败、预算耗尽继续保持 `cancelled` / `failed`。
- 失败验证、失败工具或拒绝审批为 `partial`。
- 存在任务文件变化但没有任何验证证据时仍为 `partial`，保证代码修改不会被误报完成。
- 删除类任务通过路径不存在后置条件时为 `completed`。
- 只有后台运行日志变化时，不降低任务结果。
- Git 完全不可用时仍为 `partial`，因为无法核查工作区事实。

## Boundaries

- 不做通用 shell 语法解释器；只识别命令边界上的字面量 `rm` 目标。
- 不把普通 shell 成功等同于验证成功。
- 不隐藏真实代码改动、失败验证、拒绝审批或高风险项。
- 不新增网络调用、模型调用或依赖。
- 不跑全量测试，只验证回执 builder、Git probe、新 UI 卡片与 TUI formatter。

## Verification

- Git probe 测试证明未跟踪文件被删除时状态为 `removed_untracked`。
- builder 测试证明 `rm -rf` 成功且路径不存在时回执为完成，并产生文件系统验证。
- builder 测试证明 shell 非零退出码或目标残留时不能完成。
- builder 测试证明代码变更无测试仍为部分完成。
- builder 测试证明 Naumi 调试轨迹归入后台且不降低结果。
- 新 UI 与 TUI 测试证明成功删除回执不再显示 `0/0`、逐文件刷屏和通用提交建议。
- 临时真实 Git 仓库中执行一次真实删除命令，复验工具事件到两套 renderer 的完整链路。

## Self Review

该设计没有通过放宽所有 `changes and not validations` 规则来掩盖问题；代码改动的验证门槛保持不变。shell 识别刻意只覆盖可证明的删除目标，复杂脚本会回退到原有保守行为。后台路径采用明确白名单而不是把“无法归因”的文件都隐藏，避免漏报真实改动。
