# 模型交付与最终审核协议

## 1. 领取格式

实现模型开始前必须声明：模块 ID、依赖状态、目标 commit、预计修改文件、定向测试命令、
明确非目标。未声明模块 ID 的改动不进入审核。

## 2. 实现纪律

1. 读取模块文档与直接引用的源码，不从旧路线图猜测当前状态。
2. 检查工作树；已有用户改动不得覆盖、还原或混入提交。
3. 先写失败测试并保存 RED 输出，再写最小实现。
4. 每完成一个子模块运行相应小测试；不得用全量测试掩盖不精确验证。
5. 完成模块自审：正确性、隐私、安全、并发、恢复、UX、跨平台。
6. 更新模块状态和实际验收证据；不修改其他模块状态。

## 3. 必交证据包

每个模块交付说明必须包含：

- `module_id` 与 commit hash；
- `changed_files`，区分生产、测试、文档、迁移；
- RED 命令与预期失败原因；
- GREEN 定向测试命令、通过数、耗时；
- lint/type/compile 结果；
- 至少一个非 mock 真实场景的输入、观察点和结果；
- 未运行的验证及原因；
- 已知不足、风险与后续模块；
- 工作树和远端分支状态。

## 4. 审核层级

### R1 规格一致性

- diff 是否只覆盖模块范围；接口名、状态枚举、Schema 与文档一致。
- 是否创建第二套权限、任务、事件、Store 或 Worktree 机制。

### R2 代码正确性

- 重放 RED/GREEN；检查错误路径、幂等、取消、超时、空输入、极端参数。
- 检查 secret、stdout、reasoning、路径和跨工作区隔离。

### R3 产品体验

- 从用户触发到状态、进度、选择、取消、错误、回执形成闭环。
- 中文文案说明发生了什么、为什么、下一步；窄屏和降级路径可操作。

### R4 集成与发布

- 上游/下游契约测试通过；旧数据、旧前端或旧版本有明确兼容结果。
- 仅在阶段门运行全量测试、打包矩阵、真实终端和升级/回滚演练。

## 5. 审核结论

只允许：

- `approved`：全部必需证据充分，可合并或已合并。
- `changes_required`：列出精确文件、行和复现命令；修复后重新审核。
- `blocked`：外部依赖或决策缺失，说明解除条件。
- `rejected`：实现方向违反架构/安全边界，应撤回该模块提交。

审核者不得因测试数量多而忽略错误的产品语义，也不得因 UI 好看而放过协议和恢复缺陷。
