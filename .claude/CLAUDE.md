# NaumiAgent — 项目准则与愿景

## 项目愿景

NaumiAgent 的终极目标是构建一个**能自主进化的 Agent**：
- 可以阅读、理解、修改自己的源代码
- 可以自主评估自身缺陷并提出改进方案
- 可以通过反思循环（reflective loop）持续提升能力
- 工具、策略、甚至架构决策都可以由 Agent 自己迭代

当前阶段是实现基础设施（引擎、工具、记忆、安全），逐步向自进化能力演进。

---

## 绝对红线：禁止简易实现

**每一个功能都必须臻至完美。** 不接受"先简单写一下"、"后面再补"、"核心逻辑先跑通"。

具体要求：

### 禁止的行为
1. **Prompt 套壳** — 把一个 prompt 包装成工具，实际不做任何代码层面的事情。每个工具必须有真实的、可验证的代码逻辑（静态扫描、文件操作、数据计算等）。
2. **凑合能跑就行** — 只验证 import 不报错就交差。必须用真实数据/真实场景端到端验证。
3. **一次性交付多个功能** — 把三个独立功能揉在一起写完就交。必须一个一个实现，做完一个验证一个。
4. **跳过边界情况** — 只测 happy path。错误路径、空输入、极端参数、并发场景都要覆盖。
5. **忽略用户体验** — 错误提示是技术术语、状态无反馈、操作无确认。用户看到的每一条信息都要精心设计。

### 必须做到的行为
1. **多轮自我审视** — 实现完一个功能后，必须自我审视一轮：
   - 这个实现真的做到了设计要求吗？还是只是"看起来像"？
   - 有哪些地方偷懒了？
   - 如果我是用户，我会满意这个体验吗？
   - 有没有遗漏的边界情况？
2. **诚实声明不足** — 如果当前轮次确实存在未完善的地方，必须明确列出：
   - 哪里不完善
   - 为什么当前无法完善（技术限制？依赖缺失？）
   - 后续需要怎么做
3. **端到端验证** — 每个功能必须通过以下验证链路：
   - 代码能编译（ruff check + import 测试）
   - 单元测试通过（pytest）
   - 真实场景可用（用真实数据跑一遍，不是 mock）
   - 用户体验闭环（从用户触发到看到结果，整条链路通畅）

## 核心设计原则

### 1. 每个功能必须真正实现，有真实的代码逻辑
- 工具必须做实打实的工作（静态扫描、文件读写、数据分析），不能只包装一个 prompt 丢给 LLM
- 分析类工具必须有"静态扫描阶段"收集真实证据，再用 LLM 做综合推理
- 扫描逻辑要覆盖主流场景，不能只检查两三个模式就说"扫描完成"

### 2. 一个功能一个功能实现，做完验证再做下一个
- 不要把多个独立功能揉在一起一次交付
- 每个功能独立设计、独立实现、独立测试
- 做完一个跑通一个，用真实场景验证确认可用后再做下一个
- 每个功能验证无误后立即 git commit，不要积攒多个功能一起提交

### 3. 工具双通道设计
- 所有分析/操作能力必须同时支持两种调用方式：
  - **用户手动触发**：通过斜杠命令（CLI `/chaos`、TUI `/chaos`）
  - **Agent 自主调用**：注册为 Tool，Agent 可在推理链中自主决策何时使用
- 用户命令和 Agent 工具走同一套底层 execute()，不重复实现逻辑

### 4. 中文优先
- 所有用户可见的文案、命令描述、错误提示使用中文
- 代码注释用英文（代码是国际语言）
- commit message 用英文

## 技术栈与架构

```
src/naumi_agent/
├── config/          # YAML + pydantic-settings 配置
├── model/           # LiteLLM 统一模型调用 (router.py)
├── orchestrator/    # ReAct 主循环 + 自适应规划 (engine.py)
├── tools/           # 工具系统 (base.py 定义接口, 每个 .py 一个领域)
├── memory/          # 会话持久化 (SQLite) + 长期记忆 (ChromaDB)
├── safety/          # 权限沙箱 + 预算控制 + 行为监控
├── tui/             # Textual TUI 界面
├── api/             # FastAPI REST + WebSocket
└── mcp/             # MCP 协议客户端
```

### 关键约定
- 模型通过 `ModelRouter` 统一调用，支持 tier（fast/capable/reasoning）
- 工具继承 `Tool` 基类，实现 `name`、`description`、`parameters_schema`、`execute()`
- 会话自动持久化到 SQLite，auto-title 从首条用户消息提取
- 安全层：`PermissionChecker` 控制工具执行权限，`BudgetTracker` 控制 token 预算

## 自进化路线图（未来方向）

1. **Phase E: 自我审查** — Agent 可以对自己源码执行 `/chaos` `/state` 等分析
2. **Phase F: 自我修改** — Agent 可以在沙箱中修改自己的工具代码并测试
3. **Phase G: 自我进化** — Agent 评估修改效果，通过反思循环决定是否采纳
4. **Phase H: 能力扩展** — Agent 自主设计并实现新工具，注册到工具系统
